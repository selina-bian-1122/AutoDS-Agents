# 任务拆解：build-local-multi-agent-mvp

## 1. 项目脚手架与目录基线

### [x] 任务 1.1：建立前后端与本地数据目录骨架

- 任务目标：创建可承载 React 前端、FastAPI 后端、SQLite 与本地产物目录的基础工程结构。
- 影响范围：项目根目录、前端应用目录、后端应用目录、`data/` 目录、基础配置文件。
- 实施要点：采用前后端分离结构，预留 `uploads`、`samples`、`runs` 目录；统一环境变量入口和本地开发说明。
- 验收标准：仓库结构清晰，前后端均能识别其各自入口文件与依赖定义，本地数据目录可被程序初始化。
- 验证方法：进行目录结构检查、前后端启动入口检查、基础配置自检。
- 回滚方式：移除新增脚手架目录与基础配置文件。

## 2. 后端多智能体编排 MVP

### [x] 任务 2.1：实现本地运行任务与状态机

- 任务目标：实现一次分析任务从创建、规划、编码、执行、报告到完成/失败的状态流转。
- 影响范围：后端编排模块、运行状态模型、任务接口层。
- 实施要点：定义 `Planner`、`Coder`、`Executor`、`Reporter` 的统一接口；通过轻量状态机串联；限制最大重试次数。
- 验收标准：后端可为一次任务产出完整步骤记录，并在失败时进行有限重试。
- 验证方法：执行一次成功路径校验和一次失败重试路径校验，确认状态与重试计数正确。
- 回滚方式：删除新增编排模块与任务状态模型。

### [x] 任务 2.2：实现本地执行器与产物采集

- 任务目标：在受控本地目录运行 Python 分析脚本，并回收标准输出、错误输出与图表产物。
- 影响范围：后端执行器模块、文件管理模块、运行目录生成逻辑。
- 实施要点：按运行 ID 建立隔离目录；写入脚本文件；设置执行超时；归档 stdout、stderr 与图表。
- 验收标准：成功运行时能采集文本输出与图表；失败运行时能获得错误摘要并供重试使用。
- 验证方法：执行样例脚本成功/失败两类验证，确认文件与日志落盘。
- 回滚方式：删除执行器模块和运行目录生成逻辑。

### [x] 任务 2.3：实现本地持久化

- 任务目标：将运行元数据、步骤记录与产物索引写入本地 SQLite。
- 影响范围：后端持久化层、数据模型、初始化逻辑。
- 实施要点：最小表结构覆盖 `runs`、`run_steps`、`artifacts`；启动时自动初始化数据库。
- 验收标准：创建任务后数据库中可查询到运行、步骤和产物信息，接口返回内容与数据库一致。
- 验证方法：执行接口冒烟和持久化读写校验。
- 回滚方式：移除新增数据库文件与持久化层代码。

### [x] 任务 2.4：实现 `mock` 与真实模型适配层

- 任务目标：在无真实模型配置时以 `mock` 模式跑通多代理链路，并预留 OpenAI 兼容模式入口。
- 影响范围：代理实现模块、模型适配层、环境配置。
- 实施要点：`mock` 模式返回可预测计划、代码与报告；真实模式统一经适配层调用。
- 验收标准：无 API Key 时系统可完成一次端到端任务；有配置时可切换到真实模式。
- 验证方法：执行无密钥本地演示校验，并检查模式切换逻辑。
- 回滚方式：删除适配层与 mock 代理实现。

## 3. React 工作台

### [x] 任务 3.1：实现任务提交与数据集选择界面

- 任务目标：让用户在 React 页面中输入分析需求，并上传 CSV 或选择样例数据。
- 影响范围：前端页面、表单组件、API 调用层。
- 实施要点：提供问题输入框、模式选择、样例列表、上传控件和发起任务按钮。
- 验收标准：用户可提交合法请求，前端能展示提交状态与基本错误提示。
- 验证方法：进行表单交互校验、接口联调与错误场景检查。
- 回滚方式：移除任务提交页面与相关组件。

### [x] 任务 3.2：实现运行轨迹、日志与结果展示

- 任务目标：在前端清晰展示多代理步骤状态、日志摘要、重试过程与最终报告/图表。
- 影响范围：前端运行详情页、状态轮询逻辑、结果展示组件。
- 实施要点：按步骤展示代理名称、状态、耗时、摘要；成功时展示图表与报告；失败时展示错误信息。
- 验收标准：用户能从界面判断当前代理执行到了哪一步、是否重试、最终结果是什么。
- 验证方法：进行成功任务与失败重试任务的手工联调验证。
- 回滚方式：移除运行详情视图与轮询逻辑。

## 4. 可观测性与本地体验完善

### [x] 任务 4.1：补充结构化日志与样例数据

- 任务目标：为任务执行链路增加最小可观测性，并提供开箱即用的演示数据。
- 影响范围：后端日志模块、样例数据目录、前端提示文案。
- 实施要点：日志中记录 `runId`、代理名、阶段、耗时、结果；提供至少一个样例 CSV。
- 验收标准：开发者可从日志定位一次运行链路，用户无需自己准备数据也能体验系统。
- 验证方法：检查日志字段完整性，并用样例数据跑通端到端流程。
- 回滚方式：移除新增日志字段与样例数据文件。

### [x] 任务 4.2：补充本地运行说明

- 任务目标：让他人在本地能快速启动并验证该多智能体 MVP。
- 影响范围：项目说明文档、环境变量示例、运行说明。
- 实施要点：明确前后端启动顺序、所需环境、`mock` 模式默认行为和测试路径。
- 验收标准：阅读说明后可独立在本地完成启动和基本验证。
- 验证方法：按文档自走一遍本地启动与任务提交流程。
- 回滚方式：移除新增说明文档。
