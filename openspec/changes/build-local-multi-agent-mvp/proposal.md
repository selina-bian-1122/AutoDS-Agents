# 变更提案：build-local-multi-agent-mvp

## 背景与动机

当前仓库仅包含 [Idea.md](C:\Desk D\Recruiting\Interview\Others\AS Watson\Multi-Agent-System\Idea.md)，描述了一个参考 AutoDS 思路的多智能体数据分析系统，但尚无可运行工程、项目级 OpenSpec 上下文或本地化实现。用户期望在较短时间内获得一个：

- 可在 `localhost` 启动并直接测试的多智能体系统。
- 前端使用 React。
- 不依赖云数据库，仅使用本地存储。
- 参考 AutoDS 的角色拆分与协作流，但不依赖其仓库实现。

因此，本次变更聚焦构建一个“本地可运行、可观察、可演示”的多智能体 MVP，为后续迭代提供可验证的工程基线。

## 目标与范围

本次变更的目标是交付一个本地多智能体工作台 MVP，满足以下可验证行为：

1. 用户可通过 React 前端在浏览器中发起分析任务，并查看多智能体执行过程。
2. 系统采用至少四个角色代理：`Planner`、`Coder`、`Executor`、`Reporter`。
3. 系统支持本地 CSV 文件上传或选择内置样例数据集。
4. 系统通过本地编排后端驱动多智能体状态流，并支持执行失败后的有限重试。
5. 执行结果、运行日志、图表等产物保存在本地文件系统，任务元数据保存在本地 SQLite。
6. 在未配置真实 LLM API Key 的情况下，系统仍可通过 `mock` 模式完成端到端演示，便于快速本地测试。

## 非目标

本次变更明确不包含以下内容：

- 不接入云数据库、对象存储、消息队列或 SaaS 编排平台。
- 不实现用户登录、权限体系、多租户隔离。
- 不实现生产级容器沙箱或内核级隔离，仅实现本地受控执行目录与超时限制。
- 不实现 Excel、Parquet、数据库连接器等多数据源生态，初版仅保证 CSV 与样例数据。
- 不追求 AutoDS 仓库功能完整复刻，仅借鉴其多角色协作与闭环修复思想。

## 用户故事与典型场景

### 场景一：本地快速体验

作为面试或演示使用者，我希望在本机启动系统后，上传一个 CSV，并输入“分析周末销量异常”，随后看到 Planner 规划、Coder 生成代码、Executor 执行、Reporter 总结的完整链路，以便快速验证多智能体协作是否成立。

### 场景二：无 API Key 也可演示

作为本地开发者，我希望在没有配置外部模型密钥时，也能以 `mock` 模式跑通流程并观察代理状态、日志和产物，以便调试前后端集成与状态流。

### 场景三：失败后自动修复

作为调试者，我希望当执行代码失败时，系统能够把错误信息回送给 Coder，并在有限次数内重试，以便验证多智能体闭环修复能力。

## 临时项目基线建议

在 `openspec/project.md` 缺失的前提下，本次变更建议采用如下临时基线：

- 前端：React + Vite，提供聊天面板、运行轨迹面板、数据集选择/上传、结果展示区。
- 后端：Python + FastAPI，负责任务编排、代理调用、文件管理和运行状态查询。
- 编排方式：自定义轻量状态机，不强依赖 LangGraph；保留后续替换空间。
- 数据存储：SQLite 保存任务、步骤、消息、运行状态；本地文件目录保存上传数据与产物。
- 执行器：在本地工作目录中运行受限 Python 脚本，设置超时、独立运行目录、只暴露必要输入输出。
- 模型模式：默认支持 `mock` 模式；若用户配置 OpenAI 兼容 API，再启用真实模型模式。

## 主要方案

## 1. 对外接口

后端提供最小可用本地接口集：

- 创建分析任务：提交用户问题、数据集选择或上传文件。
- 查询任务详情：返回任务状态、代理步骤、重试次数、产物列表。
- 获取任务日志：返回代理消息、执行输出、错误摘要。
- 获取样例数据集列表：便于无上传文件时快速体验。
- 下载产物：下载图表或生成的中间文件。

前端基于这些接口实现单页工作台，不做服务端渲染要求。

## 2. 数据模型与本地存储

建议最小数据模型：

- `runs`：一次分析任务的元数据、模式、状态、开始结束时间。
- `run_steps`：每个代理步骤的输入摘要、输出摘要、耗时、状态、重试次数。
- `artifacts`：结果文件名、类型、关联运行、相对路径。

本地产物目录建议：

- `data/uploads/`：用户上传数据集。
- `data/samples/`：内置样例数据。
- `data/runs/<run-id>/`：执行代码、stdout、stderr、图表、汇总文本。

回滚策略：

- SQLite 为单文件存储，可直接回滚至变更前文件快照。
- 产物目录按运行隔离，移除新增目录即可回滚。

## 3. 关键流程

主流程：

1. 用户在 React 前端输入需求并上传或选择数据。
2. 后端创建 `run` 记录并提取数据集 schema 摘要。
3. Planner 生成 3 到 4 步计划。
4. Coder 基于计划和 schema 生成 Python 分析脚本。
5. Executor 在本地工作目录运行脚本。
6. 若失败，则将错误摘要与先前代码回传给 Coder，最多重试固定次数。
7. 若成功，则 Reporter 生成业务友好的总结。
8. 前端轮询或流式刷新状态，展示步骤、日志、图表与最终报告。

## 4. 安全、权限与隐私

- 默认不上传数据到云端数据库。
- `mock` 模式下不调用外部模型。
- 真实模型模式下，仅发送 schema 摘要、任务描述和执行错误摘要，不默认发送完整原始数据。
- 执行器必须在单独工作目录运行，限制执行超时并清晰记录输入输出。
- 初版不引入登录，因此仅面向单机本地演示场景。

## 5. 性能与容量

- 目标数据规模为演示级 CSV，单文件建议控制在数十 MB 以内。
- 前端仅展示预览行、执行日志摘要与产物列表，避免一次性加载完整数据。
- 后端执行采用同步提交 + 异步轮询状态模式，确保实现简单、可调试。

## 6. 兼容性与回滚

- 优先向后兼容未来真实 LLM 模式与更换编排框架的演进。
- 目录结构与接口命名保持稳定，为后续替换数据库或执行器预留边界。
- 若本次实现不可用，可通过移除新增应用目录、SQLite 文件与 `data/` 目录回滚到初始空仓状态。

## 风险清单与缓解措施

1. 本地执行器存在安全边界不足风险。
   缓解：限制到受控目录、固定超时、最小环境变量注入，并在 proposal 中明确仅面向本地可信环境。
2. 无真实 LLM 时演示价值不足。
   缓解：提供 `mock` 模式，确保多代理状态流、错误重试和结果展示可被验证。
3. 前后端双栈实现可能超出 20 分钟窗口。
   缓解：严格控制为 MVP，只做单页 React 工作台、最小 API 与最少实体。
4. 缺失 `openspec/project.md` 可能导致实现标准不统一。
   缓解：本提案给出临时项目基线，待用户审批后作为本次实现依据。

## 决策记录（Decisions）

1. 默认使用 React + Vite。
   理由：满足用户对 React 的明确要求，启动快、脚手架轻、适合本地演示。
2. 默认使用 FastAPI 作为后端。
   理由：与数据分析和 Python 执行器天然契合，便于快速实现多代理编排与本地执行。
3. 默认使用 SQLite + 本地文件系统。
   理由：满足“仅本地，不用云数据库”的约束，同时保留结构化历史记录与产物管理。
4. 默认提供 `mock` 与 `openai-compatible` 两种代理模式，其中 `mock` 为默认。
   理由：保证无外部依赖时也能演示，多代理协作不会被 API Key 阻塞。
5. 默认错误处理为显式失败、记录结构化日志、有限重试。
   理由：符合 OpenSpec 默认决策矩阵，便于观察与调试。
